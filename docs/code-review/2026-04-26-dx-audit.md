# DX Audit: bones vs. beads

**Date:** 2026-04-26
**Scope:** Compares developer experience for **adding work** and **running** the system, head-to-head with [beads](https://github.com/gastownhall/beads) — the project bones was originally audited against (see ADR 0017).
**Audience assumed:** solo developer or small team. Multi-agent orchestration use case is called out separately.

## TL;DR

| | beads | bones |
|--|------|-------------|
| First-time setup | 9/10 | 4/10 |
| Add a task (human) | 9/10 | 2/10 |
| Add a task (agent) | 7/10 | 7/10 |
| List backlog | 9/10 | 8/10 |
| Claim + complete (agent) | 8/10 | 7/10 |
| Run a trial / experiment | n/a | 8/10 |
| Orchestrate parallel agents | 1/10 | 8/10 |
| Observe runtime state | 8/10 | 4/10 |
| Recover from failure | 7/10 | 5/10 |
| Update or close (human) | 9/10 | 2/10 |
| **Weighted overall** (daily-heavy) | **~7.5/10** | **~5.5/10** |

bones is **slick** at the things beads cannot do at all (parallel orchestration, behaviorally-validated commit propagation, slot-disjoint contracts) — and **rough** at the things beads makes trivial (single-task add/list/close from a human's hand, single-binary install).

The overall score gap is mostly the human-as-task-author gap: bones has no `agent-tasks add` and no `agent-tasks close`. The CLI surface is oriented entirely around *agents* consuming the backlog, not humans curating it. That's a fixable gap.

---

## What I did

1. Surveyed bones's actual CLI: `cmd/agent-init`, `cmd/agent-tasks`, `cmd/orchestrator-validate-plan`, `cmd/space-invaders-orchestrate`.
2. Surveyed beads' surface from `reference/beads/` and `reference/CAPABILITIES.md`.
3. Walked through 10 workflows, listed exact steps for each, scored, identified the friction.
4. Ranked improvements by `(frequency × gap-severity × feasibility)`.

---

## Per-workflow detail

### W1. First-time setup

**beads:** `cargo install beads` (or equivalent) → `bd init` in your repo dir → ready. **2 commands.**

**bones:**
1. Clone bones repo
2. Clone EdgeSync sibling repo (in a directory `../EdgeSync/` — the `replace` directive in `go.mod` requires this)
3. `cd ../EdgeSync && make leaf` — produces `bin/leaf`
4. `cd bones && make` — builds `bin/agent-init`, `bin/agent-tasks`, etc.
5. `bin/agent-init init` — creates `.agent-infra/` workspace
6. (Optional) `bash .orchestrator/scripts/hub-bootstrap.sh` to start the hub if you need one running

**6+ steps, two repos.** Plus the EdgeSync sibling-clone constraint is a fragile chicken-and-egg: bones's `go.mod` has `replace github.com/danmestas/EdgeSync/leaf => ../EdgeSync/leaf`, so anyone cloning bones outside this layout fails to build immediately.

**Score:** beads 9/10, bones 4/10.

---

### W2. Add a single task (human-driven backlog management)

**beads:** `bd add "Fix the login bug"` — done. `bd add --files src/login.go "Fix login bug" --priority high` for richer cases.

**bones:** **no CLI equivalent exists.** `cmd/agent-tasks` has subcommands `autoclaim`, `compact`, `dispatch`, `exit`, `health`, `json`, `link-ready`, `prime`, `open|claimed|closed` — but no `add`. The `agent-tasks open|claimed|closed` subcommands are *filters* on existing tasks, not creators.

To add a task today:
1. Write a Go program that imports `coord`
2. `coord.OpenHub(ctx, workdir, addr)` — bring up a hub
3. `coord.OpenLeaf(ctx, LeafConfig{...})` — bring up a leaf
4. `leaf.OpenTask(ctx, title, files)` — finally creates the task
5. Build & run that program

This is **a programmatic-only flow**. There is no human-friendly path to add a single task to an existing backlog. The CLI was written for *agents* to consume the backlog (autoclaim, prime), not for humans to curate it.

**Score:** beads 9/10, bones 2/10.

This is the single biggest DX gap. README.md positions the project as "Beads-inspired" but the human authoring story is missing.

---

### W3. Add a task (agent / programmatic)

**beads:** an agent calls a thin binary (`bd add ...`). Some boilerplate.

**bones:** `leaf.OpenTask(ctx, title, files)` is **deeper** — it ties to hold-gating, slot-disjointness, the claim/commit lifecycle. For agents it's actually the right shape. Comparable score.

**Score:** both 7/10.

---

### W4. List the backlog

**beads:** `bd ready` (actionable) or `bd list --status=open` (everything).

**bones:** `bin/agent-tasks open` works (post-ADR-0019 CLI). Output is JSON-tagged tabular text. Good but slightly less polished than beads' columnar output.

**Score:** beads 9/10, bones 8/10.

---

### W5. Claim and complete a task (agent)

**beads:** `bd update --claim <id>`; do work; `bd close <id> --reason "fixed"`.

**bones:** programmatic via `coord.Leaf.Claim` → do work → `coord.Leaf.Commit` → `coord.Leaf.Close`. The 4-step lifecycle is appropriate for the architecture (hold-gating, fork detection, sync) but heavier than `bd update --claim`. There's no human-friendly `bin/agent-tasks claim <id>` CLI.

**Score:** beads 8/10, bones 7/10.

---

### W6. Run a trial / experiment

**beads:** N/A (not its purpose).

**bones:** `OTEL_EXPORTER_OTLP_ENDPOINT= go run ./examples/herd-hub-leaf/` — one command, runs end-to-end, prints results. Phase 2 trial sweep proved this works clean from N=4 to N=100.

**Score:** bones 8/10. Genuinely slick.

---

### W7. Orchestrate parallel agents (the core bones story)

**beads:** N/A — agents share state via Dolt cell-merge but there's no built-in dispatch. Multiple agents scribbling in parallel is on the operator.

**bones:** orchestrator skill takes a slot-annotated plan, dispatches Task subagents in parallel, hub aggregates commits, validates disjoint slots. Phase 3 demonstrated 4-slot 2D and 5-slot 3D Space Invaders end-to-end.

**Score:** beads 1/10, bones 8/10. Genuinely slick, and the entire reason to use bones over beads.

---

### W8. Observe runtime state

**beads:** `bd list`, `bd ready`, `bd dep` queries. All synchronous, all readable.

**bones:** for static "what's the current backlog", `bin/agent-tasks open|claimed|closed` works. For dynamic "what's happening right now" — only OTLP + SigNoz dashboards. There's no `bin/agent-tasks watch` or `bin/agent-tasks tail` that subscribes to NATS and streams task lifecycle events to stdout.

**Score:** beads 8/10, bones 4/10.

---

### W9. Recover from failure

**beads:** edit the SQLite/Dolt DB if state is corrupt. Same as any single-process tool.

**bones:** `bash .orchestrator/scripts/hub-shutdown.sh` cleans up `bin/leaf` PIDs. NATS store and fossil files are in `.agent-infra/` — recoverable but manual. Lost in-flight commits surface as `ForkUnrecoverable` (defense-in-depth assertion). No `bin/agent-tasks gc` or `bin/agent-tasks reset`.

**Score:** beads 7/10, bones 5/10.

---

### W10. Update or close a task (human)

**beads:** `bd update <id> --priority high`, `bd close <id> --reason "fixed"`. Trivial.

**bones:** **no CLI equivalent for human use.** Same gap as W2. Programmatic only.

**Score:** beads 9/10, bones 2/10.

## What's slick about bones

1. **Parallel-orchestration story.** Hub + N leaves + slot-disjoint validator is a real architectural advantage. Phase 2 shows N=64 with sub-100ms P99; N=100 sustains 98% propagation. beads can't touch this.
2. **Trial harness as documentation.** `examples/herd-hub-leaf/` is a runnable, parameterized N-leaf stress trial that doubles as a tier-discovery tool. Phase 2 trial report turns it into evidence.
3. **Single architectural commit path.** Post-Phase-1, `(*Leaf).Commit` is the only way to write fossil. No `Coord.Commit` parallel path. No fork+merge. Easy to reason about.
4. **`make check`** is comprehensive: fmt-check, vet, lint, race, todo-check, all package tests, ~15s. Catching regressions is fast.
5. **ADRs.** 24 ADRs, all under 300 lines, documenting decisions and rationale. Beads has prose (README, AGENT_INSTRUCTIONS.md, AGENTS.md, ARTICLES.md) but not a structured decision log.
6. **Boundary asserts.** Tiger Style assertions at every public-API entry. Faults surface as panics with exact location, not silent corruption.
7. **Behavior-validated.** Phase 3 ran real LLM subagents producing integratable game code from a contract. The pipeline works.

## What's rough about bones

1. **Two-repo dependency** with `replace ../EdgeSync/leaf` directive. Anyone outside the sibling layout can't build. PR #79 is a step toward removing the replace, but it's still a manual setup hurdle.
2. **No human task CLI.** `bin/agent-tasks` lacks `add`, `update`, `close`, `claim` subcommands. Big gap vs beads.
3. **Hub-bootstrap depends on EdgeSync `bin/leaf`.** If `bin/leaf` isn't built, `hub-bootstrap.sh` fails. No auto-build.
4. **Workspace concept is half-baked.** `agent-init init` creates `.agent-infra/` but `coord.OpenHub` and `coord.OpenLeaf` take an arbitrary `workdir` argument — they don't use the workspace by convention.
5. **No live observability without OTLP.** SigNoz is great when configured; without it, runtime visibility is "tail the logs."
6. **Configuration discovery.** Env vars: `LEAF_*`, `BRIDGE_*`, `HERD_*`, `OTEL_*`, `AGENT_INFRA_*`. No central listing in README.
7. **Two binary build flows** — `Makefile` builds `bin/agent-init`, `bin/agent-tasks`; EdgeSync's Makefile builds `bin/leaf`. Two `make` calls in two directories.
8. **README is honest but unfinished.** "Working title", "Project name is not decided" suggest pre-1.0 polish. Fine for now; flagging.

## What's slick about beads

1. **Single-binary install.** One install, one `bd init`, ready.
2. **Ergonomic CLI.** `bd add`, `bd close`, `bd ready` map directly to mental model.
3. **Trivial to teach.** A new user can be productive in ~60 seconds.
4. **Self-contained state.** `.beads/beads.db` (SQLite/Dolt). No NATS, no sibling repos, no hub.
5. **`bd dolt push/pull`** for cross-machine sync without setup ceremony.

## What's rough about beads

1. **No coordination.** Two agents claiming the same task: last-write-wins or DB conflict. No commit-and-sync semantics.
2. **No real-time signaling.** Agent A finishing a task doesn't notify Agent B; B has to poll.
3. **Local-only by default.** `bd dolt push` exists but is opt-in.
4. **No commit/sync of code.** Tracks tasks only. Code lives in git, with all the merge tax that multi-agent commits imply.
5. **No fork/conflict detection.** If two agents both close, last write wins.

## Ranked improvements (highest leverage first)

### I1. Add `bin/agent-tasks add | update | claim | close` subcommands

**Frequency:** daily. **Gap:** 7 points (W2, W10 score 2/10 vs beads 9/10). **Feasibility:** small, mostly a CLI shell over existing `coord.Leaf` methods.

Each is ~30-50 lines of Go. Use the existing `cmd/agent-tasks` framework. Auto-discover the active leaf via `agent-init join` so the user doesn't pass workdir every time.

Example:

```bash
agent-tasks add "Fix login bug" --files src/login.go --priority high
agent-tasks claim <id>
agent-tasks close <id> --reason fixed
```

This single change closes the largest score gap vs beads.

### I2. Single-command bootstrap (`agent-init init` builds bin/leaf if missing)

**Frequency:** once-per-environment, but every new contributor hits it. **Gap:** 5 points (W1 4/10 vs beads 9/10). **Feasibility:** medium.

`agent-init init` should:
1. Detect or create the workspace.
2. If `../EdgeSync/leaf/cmd/leaf` is present, build `bin/leaf`.
3. Run `hub-bootstrap.sh` and write the hub URL into `.agent-infra/state.json` so `OpenHub`/`OpenLeaf` calls don't need URL plumbing.

Eliminates the "two repos, two makefiles" setup tax.

### I3. `agent-tasks watch` — live NATS lifecycle stream

**Frequency:** anytime an agent run is in flight. **Gap:** 4 points (W8 4/10 vs beads 8/10). **Feasibility:** small.

Subscribe to `coord.task.*` NATS subjects, print human-readable lifecycle events to stdout. No OTLP needed.

```
$ agent-tasks watch
[12:34:56] slot-A claimed task=fix-login-bug
[12:35:02] slot-A committed manifest=4aa81987 (3505 bytes)
[12:35:02] slot-A closed task=fix-login-bug
```

### I4. `agent-tasks status` — one-shot snapshot

**Frequency:** daily. **Gap:** 2-3 points (W8). **Feasibility:** trivial.

```
$ agent-tasks status
hub:        http://127.0.0.1:8765 (pid 12345)
nats:       nats://127.0.0.1:4222
leaves:     4 active (slot-A, slot-B, slot-C, slot-D)
backlog:    3 open · 1 claimed · 12 closed (last 24h)
last sync:  2s ago
```

### I5. Drop the `replace ../EdgeSync/leaf` directive

**Frequency:** every time anyone clones. **Gap:** see W1. **Feasibility:** depends on EdgeSync `leaf/v0.0.3` tag (PR #79 in flight).

Once tagged, drop the replace. bones builds standalone for anyone cloning.

### I6. README quickstart

**Frequency:** every new user reads it. **Gap:** see W1. **Feasibility:** small (writing).

Add a "60-second quickstart" section:

```bash
git clone https://github.com/danmestas/bones
cd bones
make
bin/agent-init init
bin/agent-tasks add "my first task"
bin/agent-tasks open
```

Six commands, working backlog. Removes the discovery tax.

### I7. Conventionalize workspace lookup in `coord.OpenHub` / `coord.OpenLeaf`

**Frequency:** every test, harness, or example. **Gap:** moderate (current `workdir` is implicit-but-arbitrary). **Feasibility:** medium.

Make `OpenHub(ctx, workdir, addr)` optionally accept `workdir == ""` and fall back to `agent-init join`'s `.agent-infra/`. Same for `OpenLeaf`.

### I8. Document configuration env vars in one place

**Frequency:** every new user. **Gap:** small but recurring. **Feasibility:** trivial (writing).

Add `docs/config.md` listing every env var, default, and consumer.

---

## Summary

bones's architectural advantages are real and validated: hub-and-leaf with EdgeSync NATS mesh, slot-disjoint orchestration, behaviorally-tested commit propagation. **What it currently can't do is welcome a human user adding a task to its backlog.**

Closing that single gap (I1: `add | update | claim | close` CLI) plus a quickstart README (I6) would lift the overall daily score from ~5.5/10 toward ~8/10 — at which point bones has both a beads-class authoring story AND its own orchestration story on top.

The other improvements (I2 bootstrap, I3 watch, I4 status, I7 workspace conv) are polish — important, but only after the authoring CLI lands. Without I1, bones is fast and powerful for agents and tedious for humans; with I1, it's fast and powerful for both.
