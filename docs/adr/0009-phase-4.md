# ADR 0009: Phase 4 — presence, glob Subscribe, admin-override Ask, reactions

## Status

Accepted 2026-04-19. Drives Phase 4 implementation of `coord.Who`,
`coord.WatchPresence`, `coord.SubscribePattern` (pending design
resolution — see Open Questions), `coord.AskAdmin`, and `coord.React`.

## Context

Phase 3 (ADR 0008) delivered chat narrowly — `Post`, `Ask`, `Answer`,
`Subscribe` on a single thread. ADR 0008 explicitly scoped-out five
concerns to Phase 4: presence, glob-pattern Subscribe, admin-override
Ask, reactions, and media payloads. ADR 0006 scoped-out closed-task
compaction to a later phase. Phase 4 takes up four of those six; media
payloads and closed-task compaction defer further.

Two structural pressures also come due. ADR 0008 flagged that "the
three-manager Coord is the last point where the 'just add another
field' pattern reads clean" — Phase 4's presence work is the fourth
substrate-backed field on `*Coord` and crosses that threshold. And the
`coord.Event` interface from ADR 0008 currently has exactly one concrete
type (`ChatMessage`); Phase 4 presence introduces a second
(`PresenceChange`), which exercises the sealed-interface design for
the first time.

## Decision

**Phase 4 scope.** Four deliverables:

1. **Presence** — JetStream KV-backed liveness registry, with `Who` and
   `WatchPresence` on `*Coord`.
2. **Admin-override Ask** — presence-aware `Ask` variant that
   distinguishes "recipient offline" from "recipient slow".
3. **Reactions** — a lightweight `React` method for marking messages
   with emoji-style feedback, persisted through the notify Fossil
   backing.
4. **Glob Subscribe** — pattern-based subscription across threads.
   Design open; see Open Questions.

**Deferred to Phase 5+.**

- **Media payloads** — binary content on chat messages. Requires
  binary-safe encoding in notify and a storage story that is cheap to
  serve; neither is on the critical path for Phase 4's coordination
  goals. Phase 5's code-artifact arrival (ADR 0004) will naturally
  absorb large-payload transport, at which point media reads as a
  special case of artifact delivery.
- **Closed-task compaction** — scoped-out by ADR 0006 and reaffirmed
  by ADR 0008. Phase 5+.

**Presence is first.** It's the load-bearing Phase 4 deliverable
because (a) admin-override Ask depends on it and (b) it crosses the
four-manager threshold that ADR 0008 called out, which forces the
substrate aggregate refactor before reactions or glob Subscribe pile on.

### Presence architecture

**Substrate.** JetStream KV bucket `agent-infra-presence`, one key per
agent (`AgentID`), value is a serialized `PresenceEntry`:

```go
type presenceEntry struct {
    AgentID    string    `json:"agent_id"`
    Project    string    `json:"project"`
    LastSeen   time.Time `json:"last_seen"`
    StartedAt  time.Time `json:"started_at"`
    // Reserved for Phase 4+: Status string (active/idle/busy).
}
```

KV bucket TTL is 3× `Config.HeartbeatInterval`, so two missed heartbeats
of slack before an agent's entry expires. The `3×` multiplier is a
convention, not a knob — adding a `PresenceTTLMultiplier` to Config
would be a latent foot-gun (operators could set it to 1 and get
flapping presence), and 3× is the published default across similar
liveness systems.

**Heartbeat loop.** `coord.Open` starts a goroutine that
`Put`s the entry once immediately and then at
`Config.HeartbeatInterval` cadence. `coord.Close` stops the goroutine
and deletes the entry (no wait for TTL expiry — clean shutdown should
be visible to peers immediately). Heartbeat failures are logged but
do not terminate the goroutine; presence drift during a substrate
blip is preferable to a Coord that stops heartbeating and silently
looks dead.

**Public surface on `*Coord`.**

```go
// Who returns the list of agents currently present in this coord's
// project. A fresh read; presence state is read-through the KV.
func (c *Coord) Who(ctx context.Context) ([]Presence, error)

// WatchPresence subscribes to up/down deltas for agents in this
// coord's project. The returned channel carries PresenceChange
// events; the closure tears down the subscription (invariant 17
// applies — idempotent close).
func (c *Coord) WatchPresence(
    ctx context.Context,
) (<-chan Event, func() error, error)
```

The public `Presence` DTO carries `AgentID`, `Project`, `LastSeen`,
`StartedAt` — every substrate-layer field translated through
`coord/types.go`, per ADR 0003.

`PresenceChange` implements the sealed `Event` interface from ADR 0008:

```go
type PresenceChange struct {
    AgentID   string
    Project   string
    Up        bool      // true = agent came online, false = went offline
    Timestamp time.Time
}

func (PresenceChange) eventTag() {}
```

This is the second concrete `Event` type; consumers must now actually
switch on the interface:

```go
for e := range events {
    switch v := e.(type) {
    case coord.ChatMessage:   // handle chat
    case coord.PresenceChange: // handle presence
    }
}
```

**Scope boundary.** The presence substrate is project-scoped, not
global. Agents in project A cannot see agents in project B. This
matches the existing Post/Ask project-prefixing scheme and avoids a
cross-project information leak.

### Substrate aggregate refactor

The fourth substrate-backed field on `*Coord` crosses the threshold
ADR 0008 called out. Refactor introduces an unexported `substrate`
struct carrying the four Managers:

```go
type substrate struct {
    nc       *nats.Conn
    holds    *holds.Manager
    tasks    *tasks.Manager
    chat     *chat.Manager
    presence *presence.Manager
}

type Coord struct {
    cfg    Config
    sub    *substrate
    mu     sync.Mutex
    closed bool
    subsActive atomic.Int32
}
```

`*Coord`'s public methods delegate to `c.sub.<mgr>`. The refactor is
mechanical — no behavior change, no public signature change, no test
change beyond what compiles. It lands in the same commit as presence
so the diff is one coherent change ("presence arrives + substrate
aggregates") rather than two ("substrate aggregates" as a standalone
noise commit).

`substrate.close()` tears down every manager in the reverse of open
order: presence → chat → tasks → holds → nc. Matches the current
`Coord.Close` sequence, now one-level deeper.

### Admin-override Ask

The Phase 3 `Ask` collapses "no recipient" and "recipient slow to
reply" into `ErrAskTimeout` — ADR 0008 §Recipient resolution documents
the trade-off. Presence lets us distinguish them:

```go
func (c *Coord) AskAdmin(
    ctx context.Context, recipient, question string,
) (string, error)
```

`AskAdmin` pre-flights the recipient against the presence bucket. If
absent, returns a new sentinel `ErrAgentOffline`, distinct from
`ErrAskTimeout`. If present, routes through the existing
`chat.Request` path — an online agent that fails to reply within the
ctx still surfaces `ErrAskTimeout`, preserving the caller's retry
loop.

The "admin-override" name is a holdover from ADR 0008's Phase 3
scope-out language. In practice it's a presence-aware Ask — any
agent can call it, not just an admin role. We do not add an admin
role in Phase 4; role-based authorization is a Phase 8+ concern.

`Ask` (the Phase 3 method) stays as-is. Adding the presence pre-flight
to the existing `Ask` would be a behavior-level change to an already-
shipped method; carrying it as a second method keeps `Ask`'s contract
stable and lets `AskAdmin` be the opt-in ergonomic upgrade.

### Reactions

```go
func (c *Coord) React(
    ctx context.Context, thread, messageID, reaction string,
) error
```

`messageID` is the notify-assigned ID of the message being reacted to
(surfaced on `ChatMessage` — this is a Phase 4 addition to the
`ChatMessage` DTO, which is source-compatible per ADR 0008's
"adding fields later is source-compatible" note). `reaction` is an
emoji-shaped string; coord does not validate the content — the
consumer decides which reactions it knows about.

Reactions are stored as a special-cased `notify.Message` — body is
`"REACT:" + messageID + ":" + reaction`, thread matches the original,
and the on-subscribe translation routes `REACT:*` bodies into a new
`Reaction` event type (third concrete `Event`) instead of delivering
them as `ChatMessage`.

This is a deliberately thin layer. It avoids a new KV bucket, avoids
modifying EdgeSync's `notify.Message` shape, and absorbs the
reaction-as-chat-record posture that ADR 0008's Fossil-is-the-history
decision already commits us to.

## Consequences

**`*Coord` surface growth.** Five new public methods: `Who`,
`WatchPresence`, `AskAdmin`, `React`, and `SubscribePattern` (once
designed). Two new public event types: `PresenceChange`, `Reaction`.
One new sentinel: `ErrAgentOffline`. The Phase 1 commitment was "coord
is the sole exported package"; Phase 4 keeps the promise — `presence`
lives at `internal/presence` alongside the other substrate packages.

**Substrate aggregation.** The `substrate` struct absorbs four of the
fields that previously lived directly on `*Coord`. Future manager
additions (Phase 7 MCP integration, Phase 8 ACL) land on `substrate`
without further line-count pressure on `Coord` itself.

**Event interface exercised.** `coord.Event` was a one-type interface
through Phase 3. Phase 4 adds `PresenceChange` and `Reaction`,
turning every `Subscribe*` consumer into a type-switch site. The
switch cost is a per-message interface check; consumers that care
only about chat can `continue` on non-`ChatMessage` events — the
EdgeSync notify substrate does not let us filter events server-side
by type, so the filter lives in the consumer.

**Heartbeat goroutine.** `Coord.Open` now starts a goroutine. This is
a new invariant surface: the goroutine must terminate on `Close`, and
Close must not return until it has. Reuses the sync.Once-guarded
close shape from invariants 7/16/17.

**Presence TTL vs. heartbeat cadence.** The 3× multiplier is fixed in
code, not Config. Operators who want tighter presence detection should
reduce `HeartbeatInterval`. Changing the multiplier itself requires an
ADR amendment (and operator communication — tightening it risks
flapping, loosening it risks stale presence).

**ADR 0008 §"just add another field" prediction validated.** That
section said the three-manager Coord was "the last point where the
pattern reads clean" and that adding a fourth manager would warrant a
refactor. Phase 4 confirms the prediction: the refactor is warranted,
and the chosen shape (`substrate` aggregate, unexported) matches the
path ADR 0008 sketched.

**Invariants 1-17 unchanged.** Phase 4 adds two new ones (18 below).
The existing seventeen apply to the Phase 4 methods with no
modification — `ctx != nil`, `AgentID` non-empty, coord-not-closed
checks all apply at every new method entry.

**Invariant 18 (new): presence heartbeat lifecycle.** The heartbeat
goroutine started by `coord.Open` must be terminated by `coord.Close`
before `Close` returns. The KV entry must be deleted (not merely
abandoned to TTL) on `Close`, unless the substrate is already
unreachable. Failure to delete drops to TTL-based cleanup and logs a
warning; it is not an error return, because a substrate-gone-away
presence-leak is recoverable on next heartbeat from the peer.

**Invariant 19 (new): presence TTL bound.** Presence entry TTL is
exactly `3 × Config.HeartbeatInterval`. Tightening this bound is an
ADR-level change. Enforced at `presence.Manager.Open` — the KV
bucket creation encodes the TTL from `cfg.HeartbeatInterval`
arithmetic, no operator override.

## Open questions

**Glob Subscribe.** ADR 0008 §Recipient resolution defers the glob
pattern design to Phase 4. The blocker: Phase 3's deterministic
thread identity (SHA-256 hash → ThreadShort) does not compose with
NATS subject wildcards. A subscription on `notify.<proj>.*` would
match every thread's subject, but the caller-supplied pattern
"chat.*" cannot be expanded to a finite set of deterministic
ThreadShorts without a thread-name registry.

Three options on the table:

1. **Separate raw-pattern API.** `SubscribePattern(ctx, pattern)`
   takes a raw NATS subject pattern (post-`notify.<proj>.` prefix).
   Caller is responsible for knowing the ThreadShort shape. Simplest
   implementation; leaks substrate detail to the consumer.

2. **Hybrid detect/bypass.** `Subscribe(ctx, thread)` detects a
   wildcard in `thread`, skips the hash, and routes to a raw-pattern
   subscription on `notify.<proj>.<thread>`. Consumer-opaque;
   asymmetric behavior on a single method is a usability trap.

3. **Thread-name registry.** A new KV bucket mapping thread names
   → ThreadShorts, updated at every `Post`. `Subscribe` with a
   pattern expands to a set of concrete ThreadShorts. Substrate-
   hidden; cost is one KV read per `Post` and one KV scan per
   `SubscribePattern`.

Option 3 is the most honest but also the most expensive; option 1 is
the cheapest but leaks substrate knowledge; option 2 is the
ergonomic middle. The ticket for glob Subscribe will pick one —
likely option 3 if the `Post` cost is acceptable, option 1
otherwise. This ADR does not decide; it records the trade-space so
the ticket does not re-litigate.

**ChatMessage ID exposure.** `React` needs `messageID` on the
`ChatMessage` DTO. The notify `Message` struct has an `ID` field;
Phase 3 chose not to surface it. Phase 4 adds it as a new field
(source-compatible). No ADR-level decision needed — field addition
is within the latitude ADR 0008 granted.

## Update (2026-04-19): glob Subscribe → option 1

Resolved in ticket agent-infra-6wv. Shipped as `SubscribePattern(ctx,
pattern)` with raw NATS subject-wildcard semantics: `*` matches every
ThreadShort under the project, `>` matches every ThreadShort plus any
hypothetical subsegments, and a literal pattern matches one specific
ThreadShort. No hash is applied — callers supply a substrate-level
pattern and the Manager forwards it as-is.

Picked option 1 over option 3 (registry bucket) because:

- The only universally useful patterns are `*` and `>`; both are
  equivalent to the existing `Subscribe(ctx, "")` project-wide path,
  so option 1 costs ~60 LoC plus a single chat-layer forwarder
  (`chat.Manager.WatchPattern`).
- The literal-ThreadShort case is a useful non-wildcard lane: a
  consumer can bootstrap a pattern subscription from `ChatMessage.
  Thread()` observed on an earlier subscription without round-tripping
  through the thread-name hash.
- Option 3's per-Post KV write and dynamic re-subscribe plumbing
  required a registry Manager, a KeyWatch on the registry bucket, and
  a fan-in of multiple NATS subscriptions per `SubscribePattern` —
  out of scope for Phase 4 without a concrete name-level use case.
- The substrate leak is minimal: `*` is a universal glob character,
  and `ChatMessage.Thread()` already surfaces ThreadShorts, so callers
  are not learning a new substrate concept — only that the pattern
  argument is matched against ThreadShorts rather than thread names.

Option 3 remains on the table for a future ticket if a name-level
pattern use case arrives. The two APIs are compatible: a registry
bucket would back a second method (`SubscribeByName(ctx, namePattern)`
or similar) without disturbing `SubscribePattern`'s substrate-level
contract.

The decision closes ADR 0009's glob-Subscribe open question and was
the last Phase 4 scope item. Phase 5 roadmap items (media payloads,
closed-task compaction, name-level pattern registry) are tracked
separately.
